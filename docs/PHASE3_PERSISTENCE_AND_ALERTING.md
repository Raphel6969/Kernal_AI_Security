Phase 3 — Persistence & Alerting (historical draft)
=====================================

Summary
-------
This draft recommends a production-ready database and outlines an actionable plan to implement persistent event storage, alerting, and the remaining todos tracked in the repository TODO list.

Status note: the current codebase already includes the always-on agent, live event flow, and opt-in remediation support. Treat the steps below as a planning reference for future durability and platform expansion work rather than the current implementation state.

Recommended database
--------------------
- Development: `SQLite` (zero-config, fast iteration, works in tests).
- Production (recommended): `PostgreSQL`.
  - Rationale: ACID durability, `JSONB` for rule/ML metadata, rich indexing, mature tooling, easy to host (RDS/CloudSQL), supports partitioning and retention policies.
- Analytics/long-term: optionally add `ClickHouse` or `TimescaleDB` if you need large-scale aggregation or time-series storage.

Why Postgres
------------
- Durable and well-understood for audit logs and security events.
- `JSONB` allows storing detection metadata (matched rules, explanations) while keeping relational indexes for core fields (timestamp, severity, classification).
- Works well with Python ORMs (`SQLAlchemy` / `SQLModel`) and migration tooling (`alembic`).

Suggested schema (core table)
-----------------------------
- Table: `security_events`
  - `id` UUID PRIMARY KEY (generated by backend)
  - `captured_at` TIMESTAMP WITH TIME ZONE (index)
  - `pid` INTEGER
  - `ppid` INTEGER
  - `uid` INTEGER
  - `gid` INTEGER
  - `comm` TEXT
  - `argv` TEXT
  - `command` TEXT (short indexable command string)
  - `detection_score` FLOAT
  - `classification` TEXT (e.g. `malicious`, `suspicious`, `benign`) (index)
  - `severity` SMALLINT (0-10) (index)
  - `matched_rules` JSONB
  - `ml_confidence` FLOAT
  - `explanation` TEXT
  - `alert_sent` BOOLEAN DEFAULT FALSE
  - `created_at` TIMESTAMP WITH TIME ZONE DEFAULT now()

Indexes & retention
--------------------
- Index on `(captured_at)`, `(classification)`, `(severity)`, and a GIN index on `matched_rules` for JSONB queries.
- For high-volume deployments: partition `security_events` by range on `captured_at` (daily/monthly) and periodically drop/archive old partitions.

-----------------------------------
0) Build always-on agent
   - Add an explicit agent runtime that starts the backend as a background service.

   - Add service launch scripts so the team does not need to manually start the backend each time.

   - On Linux, the agent can enable kernel capture; on macOS and Windows, it runs in API-only mode.

1) Design DB & schema (this document): finalize schema, indexes, retention rules. (in progress)

2) Implement DB models & migrations
   - Add `sqlmodel` or `SQLAlchemy` models under `backend/events/models.py`.
   - Add `alembic` for migrations and create initial migration that creates `security_events`.
   - Update `backend/requirements.txt` to include dependencies: `sqlmodel` (or `sqlalchemy`), `alembic`, `asyncpg` (for async Postgres), or `psycopg[binary]`.

3) Replace in-memory `EventStore` with DB-backed store
   - Implement a thin `DbEventStore` in `backend/events/event_store.py` that exposes the same API: `append()`, `get_recent(limit)`, `get_all()`, `size()`.
   - Keep an in-memory cache (LRU or small deque) for recent events to speed frontend hydration.
   - Migrate existing code paths (`backend/app.py`) to inject the DB-backed store.

4) Persistence tests (unit & integration)
   - Unit tests with SQLite in-memory for model-level behavior.
   - Integration test using a Postgres test container (or `pytest-postgresql`) to validate migrations and queries.

5) Add alerting & notifications
   - Implement `backend/alerts/` with webhook sender and adapters for Slack/Email. Use background task queue (FastAPI `BackgroundTasks` or `celery`/`RQ`) to avoid blocking the request path.
   - Add simple rules in `rule_engine` to trigger alerts (e.g. severity >= threshold, ML score > 0.9, specific matched rule).
   - Persist `alert_sent` flag and `alert_history` table if needed.

6) Automated response / remediation (cautious, opt-in)
   - Implement action queue and audit trail. Actions should be manual by default and require explicit enablement: e.g., block IP, kill PID, quarantine process artifact.
   - Log all actions to `remediation_log` table and allow rollback where applicable.

7) Scale & durability improvements
   - Add async background worker (Celery/RQ with Redis) for heavy work (ML scoring, enrichment, alerts).
   - Introduce a write-ahead queue (Redis/NSQ/RabbitMQ) for spikes; workers consume and write to DB.

8) Auth & access controls
   - Add API tokens / OAuth for the API and WebSocket. Protect endpoints with role-based permissions.

9) End-to-end tests & stress testing
   - Add `scripts/phase3_load_test.sh` to run k6 or locust against the API and WebSocket.

Developer-level commands & quick start
-----------------------------------
- Local dev (SQLite): set `DATABASE_URL=sqlite+aiosqlite:////tmp/kernal_events.db` and run the app.
- Prod (Postgres): set `DATABASE_URL=postgresql+asyncpg://user:pass@localhost:5432/kernal` and run alembic migrations.
